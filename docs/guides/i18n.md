# Internationalization (i18n)

DigitMile supports three languages across two surfaces:

| Surface | Translation system | Languages |
|---------|--------------------|-----------|
| Django admin + teacher dashboard (HTML templates) | Django `gettext` (`.po` → `.mo`) + `LocaleMiddleware` | `en`, `mk`, `sq` |
| Unity WebGL game (`DigitMile/game/`) | JSON dictionaries in `DigitMile/game/i18n/` loaded by the game client | `en.json`, `mk.json`, `sq.json` |

The two systems are deliberately separate because the game is served as static files by its own nginx container and does not go through Django.

## Configuration (already wired)

Source: `DigitMilePanel/digitmile/settings.py`.

```python
LANGUAGES = [
    ("en", "English"),
    ("mk", "Македонски"),
    ("sq", "Shqip"),
]
LOCALE_PATHS = [BASE_DIR / "locale"]
USE_I18N = True
```

`LocaleMiddleware` is registered in `MIDDLEWARE`. The language-switch endpoint is mounted at the project root: `path("i18n/setlang/", set_language, ...)` in `digitmile/urls.py`.

## Django-side workflow

Run commands from `DigitMilePanel/`. The working project already has a `locale/` directory; `makemessages` will update the existing `.po` files rather than create them from scratch.

### 1. Mark strings in templates and Python

```django
{% load i18n %}
<h1>{% trans "Teacher Login" %}</h1>

{% blocktrans with name=user.first_name %}
Hello {{ name }}, welcome back!
{% endblocktrans %}
```

```python
from django.utils.translation import gettext as _
message = _("Welcome to DigitMile")
```

### 2. Extract / update `.po` files

```bash
docker-compose exec backend python manage.py makemessages -l mk
docker-compose exec backend python manage.py makemessages -l sq
```

Produces `locale/<lang>/LC_MESSAGES/django.po`.

### 3. Translate

Edit the `msgstr` entries in the `.po` file. Example:

```po
msgid "Teacher Login"
msgstr "Најава на наставник"
```

### 4. Compile

```bash
docker-compose exec backend python manage.py compilemessages
```

Produces `locale/<lang>/LC_MESSAGES/django.mo`. Restart the backend container to pick up the new `.mo` files.

## Language switcher (templates)

The admin and teacher pages extend `digitmileapi/templates/admin/base_site.html`, which already renders a language dropdown in the top-right header. Pages that extend `admin/base_site.html` get the switcher for free.

For standalone pages (not extending the admin base), include the reusable component:

```django
{% include 'digitmileapi/includes/language_switcher.html' %}
```

The switcher posts to `/i18n/setlang/`, which stores the chosen language on the session.

## Unity game translations

Game strings live in `DigitMile/game/i18n/{en,mk,sq}.json`. The nginx container serves the `i18n/` directory with no-cache headers so translation updates surface without cache-busting. Translation loading is handled inside the Unity build; adding a new language means (a) adding a new JSON file with the full key set and (b) wiring the language option into the game UI (outside the scope of this guide).

## Adding a new language

1. Add the tuple to `LANGUAGES` in `settings.py`.
2. Run `makemessages -l <code>` and translate the resulting `.po`.
3. Run `compilemessages` and restart the backend.
4. If the game should support the language too, add `DigitMile/game/i18n/<code>.json` with keys matching the existing files.

## Troubleshooting

- **Switcher not visible** — confirm the page extends `admin/base_site.html` or includes the `language_switcher.html` partial.
- **Translations not applied** — make sure you ran `compilemessages` and restarted the backend; `LocaleMiddleware` reads the compiled `.mo`, not the `.po`.
- **Language not persisting across pages** — `set_language` writes to the session; confirm `SessionMiddleware` is enabled (it is by default).
